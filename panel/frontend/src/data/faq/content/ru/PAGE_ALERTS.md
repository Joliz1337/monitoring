# Алерты

Система уведомлений о проблемах с серверами. Триггеры следят за CPU, RAM, сетью, Load Average, TCP-соединениями и доступностью. При срабатывании шлют сообщение в Telegram.

## Что можно делать

- Подключить Telegram-бота для уведомлений
- Настроить триггеры по каждому ресурсу: CPU, RAM, сеть, Load Average, TCP, offline
- Задать пороги, время устойчивости, cooldown, окно сравнения
- Исключать конкретные серверы из отдельных триггеров
- Смотреть историю всех срабатываний с фильтрами и пагинацией
- Тестировать отправку сообщения в Telegram одной кнопкой

## Ключевые понятия

**Trigger** — правило: «если метрика превысила порог — прислать уведомление».

**Critical threshold** — абсолютное значение, при котором триггер срабатывает (например, CPU ≥ 85%).

**Spike %** — относительный скачок: «значение выросло на N% по сравнению с недавним». Полезно для серверов, которые стабильно грузят 40%: алерт придёт, когда станет 70% (скачок +30%), а не ждать пока будет 85%.

**Sustained seconds** — сколько секунд метрика должна держаться выше порога до срабатывания. Защита от коротких пиков: 30 секунд выше 85% = не алерт, 3 минуты = алерт.

**Cooldown** — минимальный интервал между повторными уведомлениями одного типа. Без него сервер с устойчивой перегрузкой будет спамить каждую секунду.

**Fail threshold (offline)** — сколько подряд неуспешных проверок связи до объявления сервера offline.

## Частые вопросы

### Как подключить Telegram?
1. Откройте @BotFather в Telegram, `/newbot`, получите токен
2. Вставьте токен в поле Bot Token
3. Добавьте бота в нужный чат (группу или лично)
4. Получите chat_id через @userinfobot или `getUpdates`
5. Введите chat_id в поле Chat ID
6. Нажмите «Test Send» — должно прийти сообщение

### Почему алерт не приходит, хотя CPU 95%?
Проверьте в порядке важности:
1. Триггер **включён** (toggle справа)
2. Сервер не в списке **исключений**
3. `sustained_seconds` не слишком большой — может метрика не держится так долго
4. `cooldown` не «съел» последнее уведомление
5. Telegram настроен и «Test Send» работает

### Spike vs Critical — когда что использовать?
**Critical** — если есть абсолютный предел, при котором система ломается (RAM 95%). **Spike** — если нормальная нагрузка стабильная, и важно ловить именно **рост** (CPU стабильно 30%, алерт при резком росте до 60%). Можно использовать оба одновременно.

### Что такое Load Average offset?
Вы сравниваете LA с числом ядер, но не точка-в-точку: например, `cores × 1.5` = «алерт, если LA больше чем 1.5 × число ядер». Так защищаетесь от лёгких пиков и ловите реальную перегрузку.

### История алертов — что в ней?
Полный журнал всех срабатываний: сервер, тип триггера, значение в момент срабатывания, время. Сортировка по типу и дате, пагинация. Хранится на уровне БД панели.

## Если что-то не работает

- «Test Send» падает → токен неверный, или бот не добавлен в чат, или chat_id не тот
- Алерт дублируется каждую минуту → увеличьте `cooldown` для этого триггера
- Некоторые триггеры не работают → они могут быть глобально **выключены** (toggle в шапке блока)
