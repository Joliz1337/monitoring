# Балансировщик: алгоритмы

Алгоритм решает, как HAProxy распределяет запросы между backend-серверами. Выбор влияет на равномерность нагрузки.

## Параметры
- **roundrobin** — по кругу: 1-й на A, 2-й на B, 3-й на C. Простой и предсказуемый, для коротких запросов (REST API, статика).
- **leastconn** — на backend с наименьшим числом активных соединений. Для долгих: WebSocket, SSE, БД.
- **source** — backend по хешу IP клиента. Один клиент всегда на одном сервере, пока их число не меняется. «Прилипаемость» без cookie.
- **first** — все запросы на A, пока A жив; упал — на B. Это не балансировка, а primary + backup.
- **static-rr** — как roundrobin, но без учёта веса. Быстрее, менее гибко.
- **weight** — сервер с weight 2 получает вдвое больше запросов, чем weight 1. Для серверов разной мощности.

## Полезно знать
- Для очень коротких запросов (ответ за 10 мс) счётчик соединений не успевает за реальностью — берите roundrobin, а не leastconn.
- **source** даёт «липкость» лишь частично: добавили или убрали сервер — хеш меняется, часть клиентов перекинет. Для настоящей привязки — cookie-based stick.
- Смена алгоритма без даунтайма: правка профиля → Sync → reload на ноде. Текущие соединения не рвутся.
- Нагрузка неравномерная — проверьте health checks, возможно один backend «в отказе».
