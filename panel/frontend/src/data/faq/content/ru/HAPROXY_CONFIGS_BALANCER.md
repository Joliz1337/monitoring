# Балансировщик: алгоритмы

В HAProxy балансировка между backend-серверами управляется алгоритмом. Выбор алгоритма влияет на производительность и равномерность распределения нагрузки.

## Алгоритмы

**roundrobin** — запросы идут по кругу: 1-й на сервер A, 2-й на B, 3-й на C, 4-й снова на A. Простой, предсказуемый. Идеален для короткоживущих запросов (REST API, статические файлы).

**leastconn** — каждый новый запрос идёт на backend с **наименьшим** числом активных соединений. Балансировка «по нагрузке». Подходит для длинных соединений: WebSocket, SSE, базы данных.

**source** — выбор backend вычисляется по хэшу IP клиента. Один и тот же клиент всегда на одном и том же сервере (пока число серверов не меняется). Дешёвая «прилипаемость» без cookie.

**first** — первый доступный backend. Все запросы идут на A, пока A не упадёт — тогда на B. Не балансировка, а «primary + backup».

**static-rr** — как roundrobin, но без адаптации к весу. Быстрее, но менее гибкий.

## Weight (вес)

Каждому backend можно задать вес. Сервер с weight 2 получает вдвое больше запросов, чем weight 1. Полезно, когда у вас серверы разной мощности.

## Какой выбрать

| Ваше приложение | Алгоритм |
|-----------------|----------|
| REST API, короткие запросы | roundrobin |
| WebSocket, долгие соединения | leastconn |
| Нужна «прилипаемость» без cookie | source |
| Один основной, остальные резерв | first |
| Разная мощность backend | roundrobin + weight |

## Частые вопросы

### leastconn точно лучше roundrobin?
Не всегда. Для очень коротких запросов (ответ за 10мс) счётчик соединений не успевает отражать реальность. Используйте roundrobin. Для долгих соединений (1+ секунда) — leastconn.

### source даёт sticky session?
Частично. Пока клиент с одним IP и пока число серверов не меняется — да. Но если добавили/убрали сервер — хэш изменится, и часть клиентов перекинет. Для настоящей «липкости» — cookie-based stick.

### Как поменять алгоритм без даунтайма?
Обновить конфиг профиля, нажать Sync → на целевом сервере HAProxy reload. Текущие соединения не рвутся, новые идут по новому алгоритму.

## Если что-то не работает

- Нагрузка неравномерная → проверьте health checks, возможно один из backend «в отказе»
- source не даёт «липкости» → добавляли серверы? хэш перекинул клиентов
