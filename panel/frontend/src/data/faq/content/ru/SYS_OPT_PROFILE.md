# Профиль оптимизаций: VPN или Panel

Два готовых набора системных параметров для разных типов нагрузки. Выбираете один раз — применяется на ноду.

## Профиль VPN

Для серверов с высоким сетевым трафиком: прокси, VPN, Xray, маршрутизаторы.

**Параметры:**
- `fs.file-max` = 2M
- `net.netfilter.nf_conntrack_max` = 2M
- Socket buffers: 128 MB
- `nofile` limit: 1M
- `nproc` limit: 1M

**Смысл:** тысячи одновременных соединений, каждое требует socket buffer и запись в conntrack. Без этих настроек сервер упирается в лимиты ядра задолго до реального потолка.

## Профиль Panel

Для серверов с лёгкой нагрузкой: панели управления, мониторинг, Remnawave, API.

**Параметры:**
- `fs.file-max` = 524k
- `net.netfilter.nf_conntrack_max` = 262k
- Socket buffers: 16 MB
- `nofile` limit: 65k
- `nproc` limit: 32k

**Смысл:** обычное приложение, пара сотен соединений в секунду. Большие буферы — только зря тратят RAM.

## Как выбрать

Одно правило: **через что идёт больше всего трафика — это VPN-нода**.

- Xray, Shadowsocks, WireGuard, OpenVPN → **VPN**
- Grafana, Remnawave панель, этот мониторинг → **Panel**
- Сомневаетесь → **Panel** (безопаснее, занимает меньше памяти)

## Частые вопросы

### Можно ли сменить профиль без перезагрузки?
Частично. sysctl применяется сразу, limits.conf — только для новых shell-сессий. Для гарантированного полного применения перезапустите сервисы (или сервер).

### Поставил VPN на панель, что произошло?
Панель съедает больше RAM (под буферы), но работает. Если сервер маленький (1–2 GB) — может быть тесно, OOM убьёт что-нибудь. Переставьте на panel.

### Поставил Panel на VPN-сервер, что произошло?
Ограниченный conntrack: начнутся сбросы новых соединений («insufficient resources»), клиенты будут жаловаться на таймауты. Симптом: рост `NF_CONNTRACK_OVERFLOW` в `dmesg`. Срочно переставить на vpn.

### Что такое nf_conntrack?
Таблица соединений ядра Linux. Каждое активное TCP/UDP соединение = запись. По умолчанию лимит ~250k, для VPN-нагрузки этого мало.

## Если что-то не работает

- Профиль применился, но изменений не видно → проверьте `sysctl -a | grep conntrack_max`, значения должны совпадать
- Limits не меняются → выйдите и зайдите в shell заново, либо рестартуйте сервис
