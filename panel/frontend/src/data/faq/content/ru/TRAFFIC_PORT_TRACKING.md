# Отслеживание портов

Добавление конкретных портов в список мониторинга трафика. Позволяет видеть, сколько данных прошло через каждый порт за период.

## Что можно делать

- Добавить порт в список отслеживания
- Видеть RX/TX и суммарный трафик по каждому порту за 30 дней
- Удалять порты из списка
- Отслеживать несколько портов одновременно

## Как это работает

Агент ноды использует iptables/nftables правила с counter для каждого отслеживаемого порта. Правила прозрачны (ACCEPT) и не влияют на фактический трафик — просто считают байты. Статистика периодически отправляется в панель.

## Типичные порты для отслеживания

| Порт | Что это |
|------|---------|
| 22 | SSH |
| 25 | SMTP |

## Частые вопросы

### Порт добавлен, но трафик 0 байт — почему?
Возможно:
1. На этом порту реально нет трафика
2. Приложение слушает на другом интерфейсе (localhost, docker bridge)
3. Правило iptables ещё не применилось (подождите минуту)

Проверка: `ss -tln | grep :443` на ноде — приложение слушает?

### Можно ли отслеживать диапазон портов?
Нет, только конкретные. Диапазоны не поддерживаются, слишком много правил.

### Как долго хранится статистика?
В истории — ~30 дней по умолчанию. Старые значения агрегируются и постепенно уходят.

### Почему цифры отличаются от `iftop` или `nethogs`?
Разные методы измерения. Наш — counter на уровне iptables, включает весь пакетный трафик до/после IP-заголовка. `iftop` считает по интерфейсу. Разница обычно в пределах погрешности.

### Нагрузка на сервер от правил?
Практически нулевая. Один counter на правило, обновляется атомарно ядром. Не заметно даже при миллионах пакетов.

## Если что-то не работает

- Порт не добавляется → возможно, лимит правил iptables достигнут, удалите неактуальные
- RX есть, а TX нет (или наоборот) → приложение односторонне, это нормально
- Цифры обнулились → сервер перезагружался, counters сбросились, статистика начинается заново
