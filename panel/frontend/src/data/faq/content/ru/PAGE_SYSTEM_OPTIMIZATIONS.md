# Системные оптимизации

Применение предустановленных наборов параметров ядра и сетевой подсистемы для ускорения работы серверов. Два профиля (VPN и Panel) под разные задачи, плюс режим NIC для обработки сетевых пакетов.

## Что можно делать

- Видеть для каждой ноды текущий профиль оптимизаций и режим NIC
- Применять профиль `vpn` или `panel` в зависимости от назначения сервера
- Переключать режим сетевой карты: Multiqueue, Hybrid или RPS
- Удалять оптимизации с ноды — откат к системным значениям по умолчанию
- Применять оптимизации массово ко всем нодам одним кликом

## Ключевые понятия

**Профиль VPN** — для серверов с высокой нагрузкой по трафику (прокси, VPN): file-max 2M, conntrack 2M, socket buffers 128MB, 1M открытых файлов. Подходит серверам, через которые идут тысячи соединений.

**Профиль Panel** — для легковесных серверов (панели, мониторинг, Remnawave): file-max 524k, conntrack 262k, socket buffers 16MB. Не тратит память на буферы, которые не нужны.

**Multiqueue NIC (аппаратный)** — современные сетевые карты имеют несколько очередей, каждая привязывается к отдельному ядру CPU. Пакеты обрабатываются параллельно. Идеально для серверов с большим трафиком. Требует поддержки NIC.

**Hybrid NIC (аппаратный + программный)** — аппаратный multi-queue на первые N ядер + RPS на остальные. Нужен, когда у NIC меньше HW-очередей, чем ядер CPU (типичный случай consumer-железа: Intel I225-V с 4 очередями на 12-ядерном CPU). Панель подсветит бейдж «рекомендуется» на кнопке Hybrid, если такой случай детектирован.

**RPS (Receive Packet Steering, программный)** — если NIC не умеет в multiqueue, ядро распределяет пакеты по ядрам программно. Медленнее аппаратного, но работает везде.

**OPT_PROFILE** — маркер-файл `/opt/monitoring/configs/OPT_PROFILE`, в котором записано имя применённого профиля. По нему панель знает, что активно.

## Частые вопросы

### Multiqueue, Hybrid или RPS — как выбрать?
Открыть страницу и посмотреть в карточке ноды строку «Hardware Multiqueue». Если `supported` и панель на кнопке Hybrid показывает бейдж «рекомендуется» — выбирайте **Hybrid** (у NIC меньше очередей, чем ядер CPU; hybrid задействует все ядра). Если `supported`, но бейджа нет — **Multiqueue** достаточно. Если `not supported` — только **RPS**. На современных серверах Multiqueue поддерживается почти всегда, но на consumer-железе ядер обычно больше, чем очередей — тогда лучше Hybrid.

### Профиль VPN поставил на панель — это плохо?
Не катастрофа, но избыточно: панели не нужен 128MB socket buffer на каждое соединение. Расход RAM будет заметно выше впустую. Переставьте на `panel`.

### Смена профиля делается налету?
Нет. Применение пишет `sysctl.conf`, `limits.conf`, запускает multiqueue/RPS скрипты и перезапускает systemd-сервис настройки. Краткий перерыв в новых соединениях возможен, но уже установленные не рвутся.

### Что произойдёт при «Remove optimizations»?
Удалятся все применённые файлы: sysctl, limits, systemd-limits, multiqueue/RPS скрипты, маркер OPT_PROFILE. Сервер вернётся к дефолтам дистрибутива. Потребуется перезагрузка, чтобы старые значения действительно ушли из ядра (или вручную применить системный sysctl).

### Ноды «устаревшие» — что значит?
Пакет оптимизаций в репозитории обновился с тех пор, как вы применяли. Нажмите «Apply», и скрипты заменятся новыми.

## Если что-то не работает

- Применение зависло → проверьте связь с нодой, посмотрите результат в статусе карточки
- Multiqueue не применяется → NIC не поддерживает, выберите RPS
- После удаления профиля сервер не вернулся к дефолту → нужна перезагрузка либо `sysctl --system`
