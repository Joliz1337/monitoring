# Sticky sessions

Привязка клиента к одному и тому же backend-серверу, чтобы его запросы всегда попадали на один и тот же сервер. Нужно для stateful приложений, где сессия хранится в памяти процесса.

## Два механизма

**Cookie-based** — HAProxy вставляет в ответ cookie (например, `SERVERID=srv2`), и в следующих запросах читает её для выбора backend. Работает только для HTTP.

**Stick-table (по IP)** — HAProxy ведёт таблицу соответствий «клиентский IP → backend». Работает для любого трафика (HTTP, TCP). Менее надёжно, если клиенты за NAT (все «одним» IP).

## Когда нужно

- Приложение хранит сессию **в памяти** одного процесса (не в БД/Redis)
- Пользователь не должен «перебрасываться» между серверами — иначе потеряет вход
- Сокет-соединения (WebSocket), которые должны идти на один сервер

## Когда НЕ нужно

- Stateless REST API
- Сессия хранится в БД/Redis/Memcached и доступна со всех серверов
- Приложение само умеет sticky через consistent hashing

## Cookie-based: как настроить

1. Включаем `cookie SERVERID insert indirect nocache` в backend
2. Каждому серверу задаём имя cookie: `server srv1 10.0.0.1:80 check cookie srv1`
3. HAProxy будет вставлять `Set-Cookie: SERVERID=srv1` в ответ
4. Следующие запросы клиента придут с `Cookie: SERVERID=srv1` и пойдут на srv1

## Stick-table: как настроить

1. Объявляем таблицу: `stick-table type ip size 200k expire 30m`
2. Добавляем правило: `stick on src`
3. Первый запрос от IP X → выбирается backend, запись пишется в таблицу
4. Следующие запросы от X → тот же backend в течение 30 минут

## Частые вопросы

### Cookie или stick-table — что выбрать?
**Cookie** — если это HTTP-приложение и клиенты принимают cookie. Надёжно, ловит только «своих» клиентов.

**Stick-table** — для TCP или когда нельзя вставлять cookie. Проблемы с NAT — все клиенты за одним IP пойдут на один сервер.

### Что если сервер, к которому «прилип» клиент, упал?
HAProxy перенаправит на другой доступный. Но сессия клиента на упавшем сервере пропала — он будет «разлогинен» и увидит ошибку.

### Sticky ломает балансировку?
Частично да. Клиенты закрепляются за серверами, и если у одного очень долгая сессия — сервер будет загружен непропорционально. Компенсируется тем, что новые клиенты всё равно распределяются.

## Если что-то не работает

- Sticky не действует → проверьте, что настроен алгоритм + cookie/stick-table, и HAProxy reloaded
- Клиенты «разлогиниваются» → возможно, сервер перезапускается чаще, чем живёт сессия
