# Health checks (проверки здоровья)

HAProxy периодически проверяет каждый backend-сервер: жив ли, отвечает ли. Если несколько проверок подряд провалились — сервер помечается DOWN и из ротации исключается. Когда снова отвечает — UP и возвращается.

## Типы проверок

**TCP check** — «подключился к порту?». Самая простая: HAProxy открывает TCP-соединение к `ip:port`. Если соединение принято — OK. Идеален для не-HTTP сервисов.

**HTTP check** — HAProxy делает `GET /health` (путь настраиваемый) и ждёт статус 200. Точнее TCP: можно ловить ситуацию «порт открыт, но приложение не работает». Требует от backend эндпоинта для проверки.

**HTTPS check** — HTTP check поверх SSL. Для backend, слушающих на HTTPS.

**send-state** — отправка «X-Haproxy-Server-State» заголовка с информацией о состоянии. Редко нужно.

## Параметры

**inter** — интервал между проверками (по умолчанию 2 секунды).

**rise** — сколько успешных подряд проверок, чтобы считать сервер UP. Обычно 2.

**fall** — сколько неудачных подряд проверок, чтобы считать сервер DOWN. Обычно 3.

**timeout check** — сколько ждать ответа проверки.

## Пример жизненного цикла

```
T=0s: check → OK
T=2s: check → OK (сервер UP)
T=4s: check → FAIL (1/3)
T=6s: check → FAIL (2/3)
T=8s: check → FAIL (3/3 → сервер DOWN, из ротации)
T=10s: check → OK (1/2)
T=12s: check → OK (2/2 → сервер UP, обратно в ротацию)
```

## Частые вопросы

### TCP или HTTP — что выбрать?
**TCP** — проще, меньше нагрузки, но ловит только «порт упал». Приложение могло повиснуть и отвечать TCP, но не обрабатывать запросы.

**HTTP** — надёжнее. Создайте в приложении `/health`, который проверяет критичные зависимости (БД, Redis) и возвращает 200 если всё ок.

### Рекомендую ли rise=1?
Нет. Если сервер «мигает» (падает/встаёт каждую пару секунд), он будет дергаться в ротации — плохо для клиентов. `rise=2–3` даёт стабильность.

### Что с fall=1?
Агрессивно: один провал → out. Если у вас сеть немного нестабильная, клиенты будут «лететь» на другие серверы слишком часто. `fall=3` — разумный компромисс.

### Backend показывает DOWN, проверка HTTP 200 — почему?
Возможные причины:
1. Проверка идёт на другой порт (опечатка)
2. Путь `/health` возвращает не 200 (403, 302)
3. Таймаут проверки меньше, чем реальный ответ
4. Firewall режет трафик между HAProxy и backend

## Если что-то не работает

- Все backend DOWN → проверьте сеть от HAProxy до backend, права firewall, сам health endpoint
- Постоянные флаппинги → увеличьте `rise` и `fall`, стабилизируется
