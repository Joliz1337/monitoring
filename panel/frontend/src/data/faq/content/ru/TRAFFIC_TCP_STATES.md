# TCP-состояния

Каждое TCP-соединение проходит через серию состояний: от установки через передачу данных до закрытия. Понимание этих состояний помогает диагностировать проблемы с сетью и приложениями.

## Основные состояния

**LISTEN** — порт открыт и слушает входящие подключения. Создаётся когда сервер вызывает `listen()`.

**SYN_SENT** — клиент инициировал соединение (отправил SYN-пакет), ждёт ответа. Нормально держится доли секунды, больше — проблемы с сетью или удалённым сервером.

**ESTABLISHED** — соединение установлено, данные передаются. Основное «рабочее» состояние.

**FIN_WAIT_1** / **FIN_WAIT_2** — локальная сторона инициировала закрытие (отправила FIN), ждёт ACK и FIN от удалённой.

**CLOSE_WAIT** — удалённая сторона закрыла соединение, локальное приложение ещё не закрыло свой конец. Должно быть кратковременно. **Накопление = баг**.

**TIME_WAIT** — соединение полностью закрыто, сокет остаётся на ~60 сек для поздних пакетов в пути. Накапливается естественно при большой нагрузке.

**CLOSING** / **LAST_ACK** — редкие состояния при одновременном закрытии.

## Что должно беспокоить

### CLOSE_WAIT растёт и не спадает
Значит, ваше приложение получает `close()` от клиента, но не вызывает `close()` само. Классический баг: забыли закрыть файловый дескриптор/socket. Со временем упрётесь в лимит открытых файлов, новые соединения начнут падать.

**Что делать:** найти приложение (`ss -tp state close-wait`), перезапустить как временное решение, пофиксить код.

### TIME_WAIT в десятках тысяч
Сам по себе не страшен до лимитов tcp tw_reuse/tw_recycle. Но если ваше приложение открывает много короткоживущих соединений (например, каждый HTTP-запрос — новое TCP) — включите HTTP keep-alive в клиенте и сервере, TIME_WAIT сократится в разы.

### SYN_SENT стабильно высокий
Проблемы с удалённой стороной или сетью: ваш сервер пытается открыть соединения, но удалённые не отвечают. Проверьте DNS, маршрутизацию, файрвол.

## Частые вопросы

### Как посмотреть TCP состояния на сервере?
```bash
ss -tan | awk '{print $1}' | sort | uniq -c
```

### Почему LISTEN в подсчётах?
Это открытые порты, ожидающие соединений. На каждый слушающий процесс — одна запись. У типичного веб-сервера с HTTP + HTTPS — 2 LISTEN.

### Можно ли очистить TIME_WAIT?
Ядро само очистит через 60 сек. Принудительная очистка через `ss -K` или `tcpkill` — только в экстренных случаях.

## Если что-то не работает

- Все состояния показывают 0 → ядро не предоставляет статистику через `/proc/net/tcp`, возможно урезанный kernel
- CLOSE_WAIT растёт → найти и пофиксить приложение, временно перезапускать
