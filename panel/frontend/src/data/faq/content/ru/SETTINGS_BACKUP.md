# Резервные копии панели

Снимок конфигурации панели, который можно скачать и использовать для восстановления. Страхуется от сбоев, позволяет мигрировать на новый сервер, даёт возможность «откатить» эксперименты.

## Что попадает в backup

- Список всех серверов (адреса, имена, папки, режим `pki_enabled`, серверные сертификаты нод)
- PKI keygen: CA cert/key и клиентский cert/key панели — критично для mTLS канала panel↔node, без них все ноды «потеряются»
- Настройки алертов, триггеры, история срабатываний
- HAProxy профили и правила
- Blocklist (глобальные правила и источники)
- Wildcard SSL сертификаты и настройки
- Учётные данные YC Account (для биллинга)
- Настройки Remnawave интеграции
- Настройки TorrentBlocker
- Общие настройки панели

## Что НЕ попадает

- История метрик серверов (CPU/RAM/сеть/трафик) — может быть очень большая, обычно миллионы точек
- Конфигурация самих нод (она живёт на нодах, не в БД панели). При восстановлении панели ноды продолжат работать с теми же сертификатами, потому что их CA восстановится вместе с keygen
- SSL-сертификат самой панели (Let's Encrypt для публичного домена)

## Как создать

1. Настройки → Backup & Restore
2. Нажмите «Create backup»
3. Панель создаст `.dump` файл с текущей датой
4. Скачайте кнопкой Download
5. Сохраните в безопасном месте (S3, облако, другой сервер)

## Как восстановить

1. Установите чистую панель **той же версии** (важно!)
2. Настройки → Backup & Restore
3. Нажмите Upload, выберите `.dump` файл
4. Подтвердите
5. Панель перезапустится с данными из backup
6. Проверьте, что серверы вернулись, алерты настроены, и т. д.

## Частые вопросы

### Как часто делать backup?
Зависит от скорости изменений. Для продакшена — ежедневно (можно через cron и API). Для dev-окружения — раз в неделю.

### Можно ли автоматизировать?
В текущей версии — нет встроенного автобекапа. Можно скриптом через API:
```bash
curl -u admin:pass https://panel/api/backup/create -o backup-$(date +%F).dump
```

### Restore на новой версии панели — нормально?
Рискованно. Схемы БД могут отличаться, миграции не всегда прогоняются на restore. Лучше восстанавливать на **той же** версии, потом обновлять штатно.

### Где хранятся backup'ы?
После создания — в БД или на диске панели (зависит от реализации). Рекомендую скачивать и хранить отдельно: если сервер панели сдохнет вместе с диском, backup тоже пропадёт.

### Сколько весит backup?
Обычно единицы–десятки МБ. Большая часть — текстовые данные (конфиги, настройки), которые отлично сжимаются.

## Если что-то не работает

- Create падает → недостаточно места на диске или права на запись
- Restore не принимает файл → неверный формат или повреждение; попробуйте другой файл
- После restore данные пустые → возможно, несовместимость схем версий
