# Firewall профили (UFW)

Централизованное управление правилами фаервола на нодах. Один профиль — набор правил, который раскатывается на много серверов сразу. Аналог HAProxy-конфигов, только для UFW.

## Что можно делать

- Создавать профили с набором правил UFW (порт, протокол, IP-источник, направление, действие)
- Задавать политики по умолчанию для входящего и исходящего трафика
- Привязывать к профилю серверы — один сервер может быть в одном профиле
- Раскатывать профиль сразу на все привязанные серверы одной кнопкой
- Видеть, какие серверы синхронизированы, а какие отстали (drift-детекция по SHA256-хэшу правил)
- Смотреть историю синхронизаций с причинами ошибок

## Ключевые понятия

**Profile** — шаблон правил UFW. Не применяется автоматически; вы явно нажимаете «Sync всех» или «Sync» на конкретном сервере.

**Default incoming / Default outgoing** — политика UFW по умолчанию: что делать с трафиком, который не подпадает ни под одно правило. `deny` — блокировать молча, `reject` — блокировать с ответом ICMP, `allow` — пропускать. Обычно `deny incoming` + `allow outgoing`.

**Action allow / deny** — разрешить или запретить трафик по правилу.

**Direction in / out** — входящее (`in`) или исходящее (`out`) направление.

**From IP / CIDR** — источник трафика. Пусто = от любого (`Anywhere`). Можно указать конкретный IP (`10.0.0.1`) или подсеть (`10.0.0.0/24`).

**Sync (синхронизация)** — раскатка профиля на ноду. Стратегия атомарная: ufw reset → set defaults → add rules → enable. Перед reset нода делает JSON-снимок текущего состояния в `/etc/monitoring/ufw_backup_<ts>.json` — если применение упало, откатывается из бэкапа.

**Node-port-guard** — защита от потери связи с панелью. Нода **отказывается применить** профиль, если в правилах нет `allow 9100/tcp in` и `default_incoming != allow`. Обойти можно только галочкой **force**.

**force** — галочка «применить даже если нет allow для порта 9100». Используется только когда вы понимаете, что делаете (например, default_incoming уже стоит на allow). Без force нода вернёт ошибку, и связь сохранится.

## Статусы синхронизации

- **synced** — профиль успешно применён, хэш совпадает
- **pending** — раскатка в процессе или ждёт начала
- **failed** — применение не удалось (нода недоступна, ошибка валидации, etc.)
- **rolled_back** — apply упал, нода откатилась к прежнему состоянию из бэкапа

## Частые вопросы

### Что значит `comment` у правила?
Это пометка для людей — в хэш не входит, на ноду через UFW не передаётся (UFW в `ufw allow ...` без явного `comment` не сохраняет описание). Чисто для удобства в панели.

### Почему добавляется правило для 9100/tcp по умолчанию?
9100 — порт API ноды, через него панель управляет сервером. Без этого правила после sync панель потеряет связь и не сможет ничего исправить.

### Что произойдёт при отвязке сервера?
Связь Server ↔ Profile в БД разрывается, но **правила на ноде НЕ откатываются**. Если нужно вернуть прежний UFW — применить другой профиль или вручную через terminal.

### Что значит «Sync всех» с force?
Раскатает профиль на все привязанные серверы, игнорируя проверку наличия правила для 9100. Опасно — если ошибётесь и забыли разрешить 9100, потеряете связь со всем пулом сразу.

### Profile pending, но sync уже выполнен — почему?
Если правило профиля изменилось после последнего sync — статус автоматически становится pending до следующего применения. Нажмите Sync чтобы перекатать.

## Если что-то не работает

- `failed: Connection timeout` → нода офлайн, проверьте сервер
- `failed: Allow rule for node API port 9100/tcp missing` → добавьте правило `allow 9100/tcp in` или включите force (на свой риск)
- `rolled_back: Rule #N failed: Invalid from_ip` → проверьте формат IP/CIDR в правиле №N (должен быть IPv4/IPv6 или CIDR, не произвольный текст)
- Серверы показывают drift после успешного sync → откройте detail профиля и нажмите Sync ещё раз; если повторяется — проблема в формуле хэша (баг)
