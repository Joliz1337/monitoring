## Серверы

Список всех серверов, которые мониторит панель. Здесь добавляют новые ноды, мигрируют существующие на общий ключ и тестируют связь. Канал panel↔node полностью зашифрован по mTLS.

## Что можно делать

- Добавить новый сервер: ввести имя, IP/домен, порт → скопировать **Installer Token** (отображается до создания) → установить ноду
- Мигрировать существующие ноды на общий ключ кнопкой **Мигрировать все** в баннере (появляется автоматически)
- Проверить соединение кнопкой «Тест»
- Включить или выключить мониторинг (кнопка power)
- Редактировать имя/адрес существующего сервера
- Удалить сервер вместе со всей его историей метрик
- Скопировать **Panel IP** для настройки firewall на ноде

## Ключевые понятия

**Installer Token** — общий base64-токен, содержащий CA-сертификат, общий серверный сертификат ноды и ключ. Один токен для **всех** нод — не нужно генерировать уникальный для каждой. Отображается в форме Add Server до нажатия «Создать». При переустановке токен не меняется.

**Shared cert** — начиная с v9.1.0 все новые ноды получают единый серверный сертификат (`SHARED_NODE_CN=shared-node`), подписанный CA панели. Nginx ноды проверяет клиентский сертификат панели, панель проверяет серверный сертификат ноды по CA — MITM невозможен.

**mTLS (mutual TLS)** — двухсторонняя проверка сертификатов. Nginx ноды требует клиентский сертификат, подписанный CA панели (`ssl_verify_client on`), а панель проверяет серверный сертификат ноды по тому же CA. Без валидных сертификатов соединение рвётся ещё на TLS-handshake.

**CA (Certificate Authority)** — корневой сертификат, который панель создаёт один раз при первой установке (ECDSA P-256, валидность 10 лет). Хранится в таблице `keygen`. Все сертификаты подписаны этим CA.

**Бейдж Old key** — нода использует старый per-server сертификат. Такие ноды можно перевести на shared cert через баннер миграции или кнопку «Мигрировать» на карточке.

**Бейдж Legacy** — нода работает по старой схеме `X-API-Key` (совсем старая версия). Требует ручной переустановки с Installer Token.

**Порт 9100** — порт nginx ноды, там идёт mTLS handshake. Должен быть открыт в firewall только для IP панели.

**Panel IP** — IP, с которого панель обращается к ноде. Разрешается в firewall ноды (UFW правило на порт 9100).

**Тест соединения** — проверяет доступность API ноды: сеть, порт, TLS/mTLS handshake, ответ `/api/version`. Если тест красный — ищи проблему снизу вверх: пинг → порт → handshake.

## Установка ноды — пошагово

1. Панель → **Серверы** → **Добавить сервер**
2. Вверху формы появится **Installer Token** — скопировать кнопкой
3. Заполнить: имя, host (IP или домен), порт (по умолчанию 9100) → **Создать**
4. SSH на сервер ноды
5. Запустить `mon → Install Node` (или `NODE_SECRET=... bash /opt/monitoring-node/deploy.sh` для неинтерактивной установки)
6. Инсталлер спросит `Вставьте Installer Token:` → вставить → Enter
7. Скрипт распаковывает сертификаты в `nginx/ssl/`, поднимает docker compose
8. В панели нажать **Тест** — должно стать зелёным «Онлайн»

> Один и тот же токен можно вставлять при установке любого количества нод — он не меняется.

## Миграция существующих нод

Если после обновления панели у серверов отображается баннер «На старом ключе серверов: N»:

1. Нажать **Мигрировать все** в баннере
2. Для нод с per-server сертификатом миграция происходит автоматически — панель отправляет shared cert на ноду через mTLS
3. Для legacy-нод (X-API-Key) откроется модалка с Installer Token — нужна ручная переустановка

Баннер исчезает когда все ноды переведены на shared cert.

## Частые вопросы

### Тест не проходит — что делать?

По шагам:

1. Пинг до IP с хоста панели — проходит?
2. `nc -zv <ip> 9100` — порт открыт?
3. `openssl s_client -connect <ip>:9100 </dev/null 2>&1 | head -20` — что говорит TLS?
   - `tlsv13 alert certificate required` — нода работает, но требует клиентский сертификат (норма — так и должно быть)
   - `unknown ca` — сертификат ноды не подписан нашим CA. Переустановите ноду с актуальным Installer Token
   - `connection refused` — nginx не запущен, проверьте `docker compose ps` на ноде
4. Docker logs ноды: `docker logs monitoring-nginx` и `docker logs monitoring-api`

### Где взять Installer Token, если форма Add Server уже закрыта?

Токен не одноразовый и не меняется. Откройте **Add Server** снова — токен будет тот же. Кнопка «Copy» сразу в форме.

### Что произойдёт при удалении сервера?

Удалится сам сервер, вся его история метрик, заметки, правила firewall и HAProxy, которые хранит панель. На самой ноде ничего не изменится — docker compose продолжит работать. Без записи в панели эта нода станет «сиротой». Чтобы восстановить — создайте новую запись и переустановите ноду с текущим Installer Token.

### Зачем нужен Panel IP, если у меня уже mTLS?

mTLS защищает данные при передаче, но не ограничивает, кто вообще может постучаться в порт 9100. Любой сканер в интернете увидит открытый порт и будет пробовать handshake. Это не опасно (без валидного клиентского сертификата соединение рвётся), но создаёт лишний шум в логах. Разрешайте 9100 только для Panel IP.

### Можно ли использовать эту ноду с двумя панелями?

Нет. Каждая нода доверяет CA конкретной панели. Если нужно, чтобы одну и ту же физическую машину мониторили две панели — это две **разных** установки ноды (на разных портах, с сертификатами от разных CA).

## Если что-то не работает

- Сервер добавился, но красный → запустите **Тест**, смотрите сообщение ошибки
- Тест TLS-ошибку (`tlsv alert unknown ca`) → сертификаты не совпадают, переустановите ноду с актуальным Installer Token
- Тест ок, метрики не собираются → убедитесь, что мониторинг включён (иконка power зелёная)
- Баннер миграции не исчезает → обновите страницу; если висит — проверьте логи backend'а
