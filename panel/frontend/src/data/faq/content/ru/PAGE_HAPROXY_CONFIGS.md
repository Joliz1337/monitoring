# Конфигурации HAProxy

Глобальные шаблоны HAProxy — одна конфигурация, применяемая к множеству серверов. Удобно, когда нужно одинаковое поведение на 10+ нодах без ручного редактирования каждой.

## Что можно делать

- Создавать профили HAProxy с набором правил балансировки
- Настраивать простые forwarding-правила (порт → target) и балансирующие (пул backend-серверов)
- Выбирать алгоритм балансировки: roundrobin, leastconn, source, first
- Включать sticky sessions (прилипание клиента к одному backend)
- Настраивать health checks (TCP или HTTP) для backend-серверов
- Править сырой конфиг вручную (для опытных)
- Синхронизировать профиль сразу на несколько серверов

## Ключевые понятия

**Profile** — шаблон конфигурации. Не применяется автоматически; вы явно выбираете, на какие серверы синхронизировать.

**Rule** — одно правило внутри профиля: слушать порт X → отправить на Y или балансировать между [Y1, Y2, Y3].

**Algorithm roundrobin** — по кругу, каждый запрос на следующий backend. Подходит для REST API и коротких HTTP-соединений.

**Algorithm leastconn** — на backend с наименьшим числом активных соединений. Подходит для долгих соединений (WebSocket, БД).

**Algorithm source** — выбор backend по IP клиента (хеш). Один и тот же клиент всегда попадает на один и тот же backend — дешёвая «прилипаемость».

**Sticky session** — настоящая привязка клиента к backend, через cookie или stick-table. Нужна для stateful приложений, где сессия хранится в памяти одного сервера.

**Health check** — периодическая проверка, жив ли backend. TCP-check = подключился к порту. HTTP-check = ответил 200 на GET /health. Параметры `fall`/`rise` задают, сколько неудачных/удачных подряд меняют статус.

**Send PROXY** — отправка заголовка PROXY v1/v2 с реальным IP клиента. Нужно, если за HAProxy стоит nginx/приложение, которому важен client IP.

**Backup server** — резерв, используется только когда основные упали.

## Частые вопросы

### roundrobin vs leastconn — когда что?
**roundrobin** — простые короткие запросы, все backend одинаково быстрые. **leastconn** — если соединения долгие (SSE, WebSocket, БД-коннекты) и сервер со 100 активными будет перегружен.

### Нужны ли sticky sessions?
Если приложение stateless (хранит сессию в БД/Redis) — нет. Если stateful (сессия в памяти одного процесса) — да, иначе каждый второй запрос попадёт на «чужой» сервер.

### Health check показывает DOWN, но сервер работает — почему?
Для TCP-check: порт открыт? Firewall не режет? Для HTTP-check: путь `/health` возвращает 200? Параметр `rise` не слишком большой?

### Synced / pending / failed — что значит?
**synced** — профиль успешно применён. **pending** — в процессе. **failed** — ошибка (битый raw config или сервер offline).

### Как безопасно катить конфиг на прод?
Сначала на один тестовый сервер → проверка → потом на весь прод-пул.

## Если что-то не работает

- Правило не работает → проверьте статус синхронизации (synced)
- Все backend показывают DOWN → health check неверно настроен
- Raw config валиден, а правила нет → несовпадение портов или опечатка в IP backend
